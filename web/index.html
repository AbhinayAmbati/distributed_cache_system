Project Overview

Consistent Hashing-Based Distributed Cache

End-to-End Architectural Guide

A self-hosted, ultra-high-performance, sharded in-memory distributed key-value store built in Go.

This system is designed for massive scale and extreme resilience, featuring consistent hashing with bounded loads, decentralized SWIM-based cluster membership, asynchronous primary-secondary replication, real-time probabilistic hot-key detection (via Count-Min Sketches), key splitting, client-side L1 LRU caching, and a built-in HTTP chaos engineering framework.

The Problem

Traditional distributed caching systems suffer from several fundamental limitations:

  • Single Point of Failure (SPOF): Systems like Redis Sentinel rely on centralized coordinators to manage failover. If the coordinator fails, the entire cluster becomes unmanageable.
  • Hot Key Bottlenecks: Standard consistent hashing maps each key to exactly one node. When a single key receives extreme traffic (the "Slashdot effect"), that node becomes a bottleneck and may crash, taking down all keys it owns.
  • Global Lock Contention: Naive in-memory stores protect the entire keyspace with a single mutex, serializing all concurrent requests and destroying throughput under parallel workloads.
  • No Automatic Failover: When a node dies in a naively sharded cluster, all keys owned by that node become permanently unreachable until manual intervention restores service.

Our Solution

This project is structured as a collection of independent, peer-to-peer cache nodes that self-organize into a cluster. The architecture ensures there is no single point of failure and no centralized coordination bottleneck. Every node is symmetric: it can accept client requests, coordinate replication, detect failures, and participate in cluster membership — all without a designated leader.

Core Components & Their Interactions

1. Client Space — The Smart Router

The client SDK is not a simple RPC stub. It acts as a smart router that maintains its own local view of the consistent hash ring. When making a request, the client:

  • Hashes the key using FNV-1a to determine the responsible primary node.
  • Routes the gRPC query directly to that node, bypassing any proxy layer.
  • Maintains a local L1 LRU cache that intercepts requests to keys flagged as "hot" by the server, eliminating network roundtrips entirely for frequently accessed data.
  • Automatically invalidates L1 entries on writes to prevent stale reads.

2. gRPC Communication Layer

The high-throughput communication portal for each node, handling two categories of operations:

  • Client-facing queries: Get, Set, Delete, Expire, and Ping — the public API surface used by application developers.
  • Internal cluster operations: Replicate (async write propagation to secondary nodes) and TransferKeys (streaming key synchronization during failover or node join events).

3. Consistent Hash Ring with Bounded Loads

The topology registry that exists on both the client and server side. Each physical node is mapped to 150 virtual nodes distributed along a 64-bit hash space. Key lookups use binary search ($O(\log n)$) to locate the nearest clockwise VNode. To prevent hot-spotting, the ring implements Google's Bounded Loads algorithm: if a node's active load exceeds 125% of the cluster average, the lookup automatically redirects clockwise to the next available node.

4. Storage Engine

A highly concurrent, sharded in-memory key-value store. The keyspace is partitioned into 256 independent shards, each protected by its own sync.RWMutex. This eliminates global lock contention and allows parallel CPU cores to read and write simultaneously across different shards. Each entry stores the byte-payload value, creation timestamp, and an expiration timestamp for TTL enforcement.

5. TTL Sweeper

Stale memory is reclaimed through a hybrid lazy-and-active expiration scheme. Lazy expiration checks the TTL on every Get and deletes expired entries on-demand. A background sweeper thread runs every 100ms, randomly sampling 20 keys per shard and aggressively re-sampling if more than 25% are expired — rapidly reclaiming memory during bulk expiry events.

6. Replication Manager

Tracks replica sets and manages asynchronous write propagation from primary to secondary nodes. Writes return to the client immediately after the primary commits locally, while concurrent goroutines replicate the data to $N-1$ secondaries in the background. On node failure, the manager automatically promotes secondaries to primaries and orchestrates streaming key synchronization to restore the replication factor.

7. SWIM Gossip Protocol

The decentralized cluster coordination layer. Runs over UDP, periodically probing random peers to maintain cluster membership. Uses a direct-ping → indirect-ping-req → suspicion → dead declaration cycle to detect node failures within 3 seconds. Membership updates are piggybacked onto existing UDP packets (up to 10 per message) and spread infection-style across the cluster. An incarnation counter mechanism prevents false-positive failure detections.

8. Hot Key Detector & Mitigation Engine

A Count-Min Sketch (4 hash functions × 4096 counters) tracks key access frequencies probabilistically with $O(1)$ memory. When a key's estimated frequency exceeds 100 accesses, two mitigations activate simultaneously: Key Splitting creates 4 sharded copies of the key to distribute read load across virtual nodes, and the server signals the client to cache the value in its L1 LRU cache, dropping read latency from ~530µs to under 1µs.

Key Features at a Glance

Zero SPOF

Fully decentralized. No master node, no ZooKeeper, no Sentinel. Every node is equal.

Self-Healing

SWIM detects failures in <3 seconds and automatically promotes replicas and re-replicates keys.

Hot Key Protection

Count-Min Sketch detects hot keys in real-time. Key splitting + L1 caching prevent node overload.

Ultra-Low Latency

256-shard lock structure, async replication, and L1 cache deliver sub-microsecond reads.

Chaos Engineering

Built-in HTTP endpoints to simulate crashes, latency injection, and network partitions.

Bounded Load Balancing

Google's algorithm caps node load at 1.25× average, preventing cascading failures under skew.

Conclusion

This Consistent Hashing-Based Distributed Cache demonstrates a production-style implementation of core distributed systems patterns using Go. By avoiding central orchestrators and relying on decentralized, peer-to-peer SWIM membership, the cluster achieves native self-healing, scaling, and fault tolerance.

The system's strength lies in the integration of its advanced mechanisms:

  1. Google Bounded Loads works in tandem with consistent hashing to prevent single-node bottlenecks by redirecting requests dynamically under high concurrency.
  2. Count-Min Sketch Hot Key Detection enables low-memory, real-time frequency estimation, immediately triggering Key Splitting to parallelize read workloads and prompting client-side L1 LRU caching to bypass the network entirely for the hottest items.
  3. Primary-Secondary replication and automatic replica failover ensure durability and high availability, even during ungraceful node terminations.
  4. Built-in Chaos APIs allow engineers to inspect the system under injected failures, latency, and partitions, validating the system's fault-tolerant properties in real-world scenarios.

This architecture offers a robust foundation for building high-scale distributed caches, rate limiters, or state stores that require predictability, low latency, and absolute horizontal scalability.
