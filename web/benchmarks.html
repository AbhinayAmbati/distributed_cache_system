Cluster & Storage Performance Metrics

Empirical benchmarks collected under high concurrent operations

Test Environment

All benchmarks were executed locally on a single machine to eliminate network variance. The test environment:

  • CPU: 11th Gen Intel(R) Core(TM) i5-11400H @ 2.70GHz (6 cores, 12 logical threads)
  • Cluster: 3 cache nodes running on 127.0.0.1 (ports 7001, 7002, 7003)
  • Configuration: 150 virtual nodes per physical node, replica count = 3, max load factor ε = 0.25
  • OS: Windows, Go 1.25

Benchmark Note: All measurements were performed on a single machine using localhost networking to isolate software overhead and ensure reproducibility. Absolute throughput and latency will vary in multi-machine deployments depending on hardware, network conditions, workload characteristics, and system configuration.

Why These Benchmarks Matter

A distributed cache is only as strong as its weakest layer. Performance bottlenecks can hide at multiple levels:

  • Network serialization: gRPC encoding/decoding overhead on every request.
  • Lock contention: Global mutexes that serialize concurrent access to shared memory.
  • Hash ring lookup: CPU cost of binary search through virtual node tables.
  • Hot key amplification: A single key receiving disproportionate traffic can saturate one node while others sit idle.

Our benchmarks are designed to stress each of these layers independently and together, measuring how the system performs under realistic concurrent workloads.

Test 1: Live 3-Node Cluster Load Test

Method: We ran 50,000 operations (20% writes, 80% reads) across 8 concurrent goroutine workers against the live 3-node cluster. Keys were drawn from a pool of 1,000 unique items. The test measured end-to-end throughput including gRPC serialization, network loopback, hash ring lookup, shard selection, and storage engine access.

Test 2: Hot Key Detection & L1 Caching Latency

Method: A single key was written and then read 150 times in a tight loop with L1 caching enabled. The first 100 reads travel over gRPC to the server. On the 100th read, the server's Count-Min Sketch flags the key as hot and returns IsHot = true. From the 101st read onward, the client intercepts at its local L1 LRU cache. L1 cache latency (0.53 µs) was measured via Go's testing.B benchmark framework (BenchmarkZipfianWorkload) for sub-microsecond accuracy.

Key Insight — Parallel Write Speedup: Sequential writes run at 1.03M/s (969.0 ns/op), but parallel writes across 12 threads achieve 2.27M/s (440.5 ns/op). Conclusion

The empirical results from our benchmark suite validate three major design decisions in the cache architecture:

  • Lock Sharding Eliminates Contention: Dividing the keyspace into 256 independent shards allows multi-threaded storage operations to execute in parallel, delivering up to 5.45 Million reads/sec and 2.27 Million writes/sec without lock bottlenecks.
  • Mitigation Strategy Prevents Node Overload: The Count-Min Sketch successfully detects high-frequency keys in real-time. By promoting hot keys to L1 caches, we achieve a 995x latency reduction (from 527.5 µs down to 0.53 µs) and increase SDK throughput from ~8.9K QPS to 1.88M QPS.
  • Bounded Loads Ensure Node Health: Implementing Google's Bounded Loads consistent hashing ensures that skew cannot collapse single nodes. Under heavy load, requests are distributed clockwise to keep physical server load capped at 1.25x the cluster average.

These benchmarks prove that combining decentralized cluster membership with smart local client routing creates a highly resilient system that scales gracefully and protects itself from common traffic spike failures.
